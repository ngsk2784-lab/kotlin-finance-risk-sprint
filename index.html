48-hour flow / 진행 방식

범위를 정한 뒤
시계를 시작합니다.

SOW 승인, 결제, 권한 있는 접근, 서면 범위, 슬롯 결정 규칙과 빌드·테스트 전제가 모두 확인된 시점부터 48시간을 계산합니다.

  1. 00H
    착수 전제 확정

    SOW·결제·권한·서면 범위·슬롯 규칙·빌드/테스트 준비 상태를 확인합니다.

  2. 04H
    정적 스캔

    고정된 7개 체크로 재현 가능한 결과를 생성합니다.

  3. 24H
    맥락 검토

    Critical 우선으로 중복 제거한 최대 30개 후보의 호출부와 설정을 읽고 오탐 가능성·영향도를 정리합니다.

  4. 48H
    납품·개선 슬롯

    경로 A 적격 시 패치·재스캔, 아니면 플레이북·인수 테스트 계획, 0건이면 예방적 검증 계획을 전달하고 30분 리뷰를 잡습니다.

Honest boundaries / 하는 일과 하지 않는 일

프리플라이트의 경계를
미리 명확히 합니다.

✓포함하는 것

결정론적 정적 휴리스틱으로 전체 기계 후보를 기록하고, Critical 우선으로 중복 제거한 최대 30개 후보를 사람이 코드 맥락에서 검토합니다. 30개 이하면 전부 검토하며, 초과분은 미검토 기계 후보로 구분합니다. 개선 슬롯은 사람이 확인한 유효 Critical, 서면 합의한 작은 수정 범위, 재현 가능한 비프로덕션 빌드·테스트가 모두 있을 때만 사람이 소규모로 수정하고 재스캔합니다.

×포함하지 않는 것

  • 런타임·부하 테스트 또는 침투 테스트
  • CVE·의존성 취약점 스캔
  • 전체 법률·규정 감사
  • 컴플라이언스 준수 보장·인증
  • 자동 수정 또는 광범위한 리팩터링

특히 재시도·서킷브레이커 체크는 WebClient, RestTemplate, @FeignClient 중심의 제한된 파일 패턴만 다룹니다.

FAQ / 자주 묻는 질문

시작 전에
확인할 내용.

48시간은 언제부터 계산하나요?+

SOW 승인, 결제, 합법적이고 작동하는 저장소 접근, 30,000 LoC 이하 서면 범위, 슬롯 결정 규칙과 빌드·테스트 준비 상태를 모두 확인한 시점부터 계산합니다. 세부 수정 범위 승인 지연, 접근 장애, 자료 누락이나 범위 변경 같은 고객 측 차단이 생기면 시계를 멈추고 새 납기를 서면 합의합니다.

0건이면 안전하다는 뜻인가요?+

아닙니다. 0건은 고정된 7개 정적 체크에서 후보가 발견되지 않았다는 뜻일 뿐, 보안·안전·규정 준수를 인증하지 않습니다.

Java나 다른 언어도 가능한가요?+

현재 런칭 범위는 Kotlin/Spring 백엔드입니다. 다른 언어나 프레임워크는 이 제안의 고정 범위에 포함하지 않습니다.

코드도 직접 고쳐주나요?+

스캐너가 자동 수정하지는 않습니다. 사람이 확인한 유효 Critical, 서면 합의한 작은 수정 범위, 재현 가능한 비프로덕션 빌드·테스트가 모두 있으면 소규모 PR 또는 패치와 해당 범위 재스캔 1회를 제공합니다. 하나라도 없으면 최우선 유효 발견의 코드 수준 개선 플레이북과 인수 테스트 계획으로 대체합니다. 유효 발견이 0건이면 가짜 발견 대신 예방적 검증 계획을 제공하며, 확인되지 않은 수정 결과를 만들지 않습니다.

무료 샘플에 비공개 저장소를 보내도 되나요?+

무료 샘플은 공개 저장소만 받습니다. 비공개 코드나 자격 증명을 이메일에 첨부하지 마세요.

외부 AI·모델 서비스에 코드를 보내나요?+

사용 여부, 제공자, 처리 목적, 보존·학습·하위처리 조건을 SOW에 적고 고객의 사전 서면 승인을 받은 경우에만 합의 범위의 코드를 전송합니다. 승인과 처리 조건 확인이 없으면 외부 AI·모델 서비스에 코드를 입력하지 않습니다.
